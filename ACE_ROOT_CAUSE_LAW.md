# 阿策铁律 - 第8条：根治律

> 生效日期：2026-06-01
> 级别：**强制执行**，违反即视为任务未完成
> 触发场景：遇到bug/故障/异常时，阿策提出降级方案或绕路方案

---

## 核心原则

**故障面前，修根因是唯一正道。降级是最后手段，不是第一反应。绕路是偷懒，不是解决方案。**

---

## 一、禁止行为

### 1.1 降级三连禁

| ❌ 禁止的降级话术 | ❌ 为什么错 | ✅ 正确做法 |
|------------------|-----------|-----------|
| "API不通，我改成RSS抓取吧" | 根因是网络/DNS问题，不是API问题 | 排查网络 → 修通API → 恢复原架构 |
| "这个函数报错，我删掉这个功能" | 功能被砍不是修复 | 定位报错原因 → 修函数 → 功能完整保留 |
| "模块有冲突，我先用简单版替代" | 简单版=阉割版，能力退化 | 排查冲突根因 → 修复兼容性 → 全功能运行 |
| "这个依赖装不上，我换成别的库" | 换库=重写=新bug | 解决依赖安装问题 → 保持技术栈一致 |
| "时间不够，先这样凑合用" | 凑合=永远凑合 | 报风险+给修复排期，不接受降级当终态 |

### 1.2 绕路三连禁

| ❌ 禁止的绕路操作 | ❌ 为什么错 | ✅ 正确做法 |
|------------------|-----------|-----------|
| 报错信息看不懂 → try-except吞掉异常 | 错误被掩盖，问题潜伏 | 看懂报错 → 修根因 → 让异常正确抛出或处理 |
| 某个字段取不到 → 写死默认值 | 数据错误被静默忽略 | 排查为什么取不到 → 修数据源/逻辑 |
| 两个模块冲突 → 注释掉其中一个 | 功能丢失 | 排查冲突原因 → 修接口/改逻辑 → 两模块共存 |
| 路径找不到 → 换一个路径硬编码 | 换了还会找不到 | 排查路径生成逻辑 → 修路径拼接 → 全环境通用 |

## 二、强制流程：故障响应三步法

遇到任何bug/故障/异常，**必须按顺序执行**：

```
Step 1: 根因定位（必须完成）
  → 复现问题
  → 查看完整报错信息（不许截断，不许跳过）
  → 定位到具体代码行/配置项/网络节点
  → 输出："根因是___，位于___"

Step 2: 根因修复（必须完成）
  → 在根因位置直接修复
  → 验证修复有效（按第7条验证律执行）
  → 输出："修复方案是___，验证结果___"

Step 3: 降级评估（仅当Step 2确认无法修复时）
  → 必须书面说明为什么无法修复（时间不够不算理由）
  → 必须标注降级的影响范围和能力损失
  → 必须给出恢复原架构的具体计划和时间点
  → 必须获得刘老板批准后才可执行降级
```

### 流程判定树

```
遇到问题
  ├─ 能定位根因？ → YES → 能修复？ → YES → 修复，结束
  │                            └─ NO → 需降级？ → 提交降级申请 → 等批准
  └─ 不能定位？ → 继续排查，禁止跳到降级
        └─ 排查超2小时仍无法定位 → 上报刘老板，由刘老板决定下一步
```

## 三、降级申请格式

**降级不是你说降就降。必须书面申请，格式如下：**

```
【降级申请】
问题：[原始问题描述]
根因：[已定位的根因，证明你排查过]
已尝试修复：[列出至少2种尝试过的修复方案及失败原因]
降级方案：[你要降级成什么]
能力损失：[降级后会丢失什么功能/精度/性能]
恢复计划：[何时、如何恢复原架构，具体到步骤]
风险评级：[P0/P1/P2]
```

**未按此格式申请的降级，一律驳回。**

## 五、违规处理

| 违规次数 | 处理 |
|----------|------|
| 第1次 | 警告，要求回到Step 1重新执行根因排查 |
| 第2次 | 该任务标记为未完成，重新执行且禁止降级 |
| 第3次 | 暂停独立开发权限，所有改动必须经阿算审查后才能提交 |

## 六、与第7条（验证律）的联动

- 第7条管"做没做对"，第8条管"做没做对的事"
- 降级方案也要验证——验证你确实修不了根因，而不是偷懒
- 绕路方案也要验证——验证绕路后的能力损失和副作用
